```text
i set up 5 MCP servers and a markdown skill, watched Claude work through 3 strategies on real Polymarket data, and the agent didn't trade any of them. that's not the failure, that's the result.
i wrote strategy_validator.md, a markdown skill that runs every hypothesis through 5 checks before backtest.
below: full setup, real experiment, every hypothesis i ran and where each one died.
Why Polymarket breaks AI trading agents
the academic data on AI traders in prediction markets is brutal. march 2026, Harvard / Arcada Labs ran 6 frontier LLMs on real prediction market accounts. $10,000 each. 57 days. on Kalshi all six lost money, between 16.0% and 30.8%. on Polymarket the same cohort averaged -1.1% - barely negative, but nobody found edge. a separate paper-trading cohort produced one outlier at +6.02% over 3 days. that's the best result anyone got, in any cohort, on any platform.
separate analysis of 222 million Polymarket trades, by Della Vedova (2026), decomposes returns into two independent components - forecasting accuracy and execution quality. the finding:
no trader type beats the price-implied accuracy benchmark; execution timing, not information, determines profitability.
retail traders pick winners 51.3% of the time and still lose $79M as a group. automated traders hit coin-flip accuracy and earn $133 M. the gap is execution, not insight.
so any agent that wins by being a better predictor is fighting the wrong battle. the layer that decides what's worth trading at all matters more than the strategies themselves.
that's the layer this article is about.
The system - 5 MCP servers
the system has 5 local MCP servers connected to Claude Desktop. MCP is Model Context Protocol, the standard for letting agents talk to external tools - filesystems, APIs, custom code. each server is a small program that exposes specific tools to the agent.
they look modular but they only matter together.
memory - holds what gets learned across sessions. without it, every new chat starts blind. observations, verdicts, rework history all get saved under named keys.
news - pulls current events. lets the system check what's happening before formulating a hypothesis.
polymarket-research - the heart of the system. market data, backtest engine on nautilus_trader (an open-source backtesting engine), strategy_validator.md, pre-flight observation tool, iterative rework loop. 11 tools total.
polymarket-paper-trader - 30 tools. runs paper trades against live order books on real Polymarket. bid-ask, fill simulation, fees, all real. no live capital, but everything else is.
adversarial-validator - runs 3 rounds of attacks on every hypothesis that survives the validator. microstructure, behavioral, regime-shift. each round must be answered with concrete numbers.
86 tools across 5 servers. but the actual product isn't the servers. it's strategy_validator.md and the discipline it enforces. the servers are wiring. strategy_validator.md is what they wire to.
The discipline pipeline
the pipeline runs every hypothesis through 7 stages. each stage filters out specific failure modes before the next one starts.
pre-flight observation - look at the data before formulating anything. wallet behavior, news lag, structural mispricings. the goal: ground every hypothesis in something visible in the data, not in a trading template.
hypothesis formulation - 2-3 hypotheses, each tied to a specific observation by memory key. entry, exit, sizing, target market, all explicit.
strategy_validator.md - 5 universal criteria + 6 Polymarket-specific failure modes. each hypothesis gets one verdict: TEST_IT, REWORK, or DROP.
the 5 criteria:
1. clarity - can the strategy be turned into code in one pass?
2. testability - does the data in the backtest engine cover it?
3. edge significance - does expected edge exceed friction (fees + spread + slippage)?
4. risk - max correlated drawdown, position cap, worst-case scenario?
5. new vs baseline - is this different from a simple mean-revert / momentum / favorite-bias play that's already priced in?
the 6 known Polymarket failure modes (any hypothesis matching one gets DROP):
1. long-tail mean reversion below the friction floor - fading spikes on <15¢ candidates where 1¢ tick = 6.7%+ friction
2. forecasting-accuracy edge - being right doesn't predict positive PnL (Della Vedova, 2026)
3. latency arbitrage via MCP - 2.7-second windows can't be captured by 100ms-latency tools
4. news-stale price arbitrage without a historical news index - structurally untestable
5. copy-trading without lag adjustment - top wallet to follower delay eats the edge
6. multi-instrument cross-market state - backtest engine is one-strategy-per-instrument by design
these failure modes weren't discovered by the validator. they were lessons learned from prior experiments and prior public research, codified as a checklist so the next hypothesis doesn't relearn them by losing money. the framework is institutional memory, not insight.
adversarial pass - 3 rounds of attacks on every hypothesis that gets TEST_IT. microstructure (does edge survive fees and spread?). behavioral (who is the counterparty?). regime-shift (what kills it in another regime?). each round must be answered with concrete numbers, not opinions.
iterative REWORK - if validator returns REWORK, up to 3 iterations are allowed to address what's flagged. v2 must change something concrete from v1. after iteration 3, force-DROP if not promoted.
backtest - only hypotheses that survive validator + adversarial reach nautilus_trader. $1000 capital, max 5 markets per run. pre-committed parameters only - any undeclared filter is a discipline gap.
paper trading - only backtest survivors with positive metrics get deployed. $1000 paper capital per hypothesis. live order books on real Polymarket.
The experiment
the pipeline ran end-to-end on real Polymarket data, starting with pre-flight observation.
the most interesting finding came before any hypothesis. on Seoul mayoral, Peru presidential, and Brazil presidential markets - all multi-candidate events with 8+ enumerated candidates - the sum of YES prices across siblings was not greater than 1. it was much less than 1. Seoul: 0.0165. Peru: 0.0025. Brazil: 0.003.
the textbook coupled-market arbitrage assumes Σ > 1. real Polymarket goes the other direction. the named candidates are pinned at min-tick. 98-99% of the mass is on the unenumerated "Other". reaching for textbook arb on these markets is fitting noise.
three hypotheses got generated. then the pipeline ran each.
H1, the spike-fade ladder on locked-floor candidates - placing layered limit sells on price spikes for candidates pinned at min-tick. validator returned REWORK on 4 separate issues: correlated exposure cap violated (75% on a single event), no real-time regime-shift detection, NO-side framing missing, spike frequency unverified.
H2, the NO-side theta harvest on collapsed-bimodal favorites - buying NO on favorites whose price distribution had stabilized at near-1.0. mid-validation, the reasoning loop output flagged a contradiction in its own arithmetic:
Wait - this isn't theta. It's a 32x lottery payoff with low probability of YES upset. I conflated capital-at-risk with face value of position.
then flagged something worse - the filter selected for markets where the favorite was correctly priced. if a favorite is correctly priced at 0.97, buying NO at 0.03 has EV ≈ 0 minus fees. lottery ticket at fair odds. the filter was anti-edge. validator returned DROP.
H3, the cross-candidate spike correlator. i caught this one myself and didn't put it through the pipeline - it hit the same architectural blocker from a prior phase:
Re-formulating into yet another peer-fetch-at-init variant is the kind of hypothesis-laundering the article warns against - same blocker, different costume.
H1 went through iterative REWORK. v2 addressed all 4 issues: 3-event diversification, regime-kill at YES > 0.15, NO-side framing explicit, eligibility filter declared. validator returned TEST_IT.
then 3 adversarial rounds.
microstructure: survived. concession:
the hypothesis survives microstructure ONLY if it explicitly assumes maker fills on both legs.
regime-kill exits become taker fills (forced market orders that pay fees instead of earn rebates), eating round-trip edge.
behavioral: survived. concession:
the hypothesis must explicitly accept that it cannot distinguish retail from informed flow in real time.
regime-shift: survived. concession:
the strategy is explicitly a late-stage / lock-confirmed regime strategy. effective N of 10-30 means this is evidence the strategy is consistent with edge, not proof of edge.
backtest. 666 trade ticks across 3 markets. 0 fills. the pre-committed kill criterion was zero or near-zero fills, and that triggered. no rework, no parameter chasing.
no hypothesis reached paper trading.
What this means
zero deployments to paper trading. that's not the system failing. the run produced 4 concrete artifacts that wouldn't have existed otherwise.
Σ(YES) finding. on multi-candidate Polymarket events, the textbook arbitrage formula points the wrong direction - Σ is far below 1, not above. pre-flight pulled this out of the data before any hypothesis got formulated. anyone reaching for coupled-market arb on these markets is now warned, including future me.
H1 v1 → v2. validator returned REWORK on 4 separate issues. v2 addressed all 4 - 3-event diversification, regime-kill threshold, NO-side framing, eligibility filter declared. validator returned TEST_IT. the iterative loop did exactly what it was built to do: turn a half-formed idea into something testable.
H1 adversarial characterization. 3 rounds of attacks, each survived with an explicit concession. the strategy is now characterized: maker-fill dependent, late-stage / lock-confirmed regime only, can't distinguish retail from informed flow in real time. that's not a failed hypothesis - that's a hypothesis with a precise edge profile. when the right market shows up, the strategy has a chance.
H2 caught early. the filter was selecting against the edge it was supposed to find. a real bug in the reasoning, flagged before any code ran. capital saved by the validator alone.
zero paper trades. but every layer fired correctly. each one filtered out something specific, and the math is visible on every step.
that's the point. discipline isn't refusing to trade. it's knowing which layer caught what, and having the answer in your hand before any capital moves.
the layers don't have to be about Polymarket. the framework - pre-flight, validator, adversarial, iterative rework, pre-committed kill criteria - applies to any decision an agent is about to make with consequences. investments. product hypotheses. deal screening. security incidents. medical decisions.
strategy_validator.md is the discipline layer for trading. the next version of this file is going to be domain-agnostic.
What didn't work
three things broke during the experiment. all three are useful to know.
the news MCP server hung mid-session on get_top_news and get_news_by_keyword. the workaround was to skip news-based observations entirely. one of the 5 servers being unstable is the reality of running this stack today.
the Polymarket data-api endpoints for individual wallet trade history were not whitelisted in the sandbox. couldn't pull SecondWindCapital's exact trade trace, only the categorical aggregate. tape patterns on likely-candidate markets had to substitute.
a pre-flight check listed but not executed is a discipline gap, not discipline. the pre-commit included this eligibility filter:
≥10 prints with YES > 0.02 in past 90d per market
the scan was never actually run before backtest. running it would have flagged engine data sparsity earlier - the same problem that produced the 0 fills outcome.
this is a hole in the experiment itself. the article argues that pre-committed checks beat ad-hoc ones, and one of the pre-committed checks didn't get executed. declarations of intent are not verified pre-conditions. the same rule has to apply to the system running the framework, not just to the strategies the framework is judging.
Next steps

more adversarial dimensions. current 3 rounds cover microstructure, behavioral, regime-shift. missing: sample size, survivorship bias, data quality. each one kills hypotheses that pass the first 3.
```
