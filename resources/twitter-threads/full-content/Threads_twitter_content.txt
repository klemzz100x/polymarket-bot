def bayes_update(prior, likelihood_if_true, likelihood_overall):
    posterior = (likelihood_if_true * prior) / likelihood_overall
    return round(posterior, 3)

def detect_insiders(prices_df, window=12, k=3.0):
    prices_df['H']     = prices_df['price'].apply(entropy)
    prices_df['dH']    = prices_df['H'].diff()
    prices_df['sigma'] = prices_df['dH'].rolling(window).std()
    prices_df['alert'] = np.abs(prices_df['dH']) > k * prices_df['sigma']
    return prices_df[prices_df['alert']]

Part 1: Why Independence Fails and Where Markov Chains Begin
Before you can build anything with Markov Chains, you need to understand the fundamental problem they were designed to solve.
Most basic probability models assume independence. Each event is treated as though it has no connection to what came before it. Roll a die. The outcome of roll 100 has nothing to do with roll 99. Each roll is completely independent.
Markets do not work this way.
Imagine you are modeling a portfolio of loans. Each loan at any point in time can be in one of four states: current, 30 to 59 days late, 60 to 89 days late, or 90 plus days late. You want to model how this portfolio evolves over time. So you try the simplest approach. You model each month independently. You draw from each state's historical distribution.
One month later, your model says that some loans that were current last month are now 90 plus days delinquent.
That is mathematically impossible. A loan cannot jump from current to 90 plus days late in a single 30-day period. The naive independence assumption produced a model that violates reality.
This is exactly the problem Markov Chains solve. Instead of assuming each step is independent of everything that came before it, a Markov Chain introduces local conditional dependence. The next state depends on the current state. Not on everything that happened ten steps ago. Just on where you are right now.
Formally, a sequence of random variables X₀, X₁, X₂, ... is a Markov Chain if it satisfies the Markov property:
P(Xₙ₊₁ = s | X₀, X₁, ..., Xₙ) = P(Xₙ₊₁ = s | Xₙ)
The probability of the next state depends only on the current state, not on the entire history. This single property is what makes Markov Chains simultaneously tractable and powerful. They capture the essential dependency structure of a process without requiring you to track the entire history.
For financial markets, this translates directly. The probability that the market will be in a bull regime next month depends on whether it is in a bull, bear, or sideways regime this month. Not on what happened two years ago. The current state carries all the relevant information you need to forecast the next state.
The Markov property states that the probability of the next state depends only on the current state, not on the full history of past states.
This is not a simplification made for convenience. It is a mathematically principled assumption that, when applied correctly, produces models that are dramatically more accurate than naive independence while remaining computationally tractable.
Part 2: Building the State Space and Transition Matrix
The first practical step in building a Markov Chain trading model is defining your states. This is more important than most practitioners realize.
For financial markets, common state definitions include:
Volatility regimes: Low volatility, medium volatility, high volatility defined by rolling realized volatility thresholds. 

Trend regimes: Bull, bear, sideways defined by the position of price relative to moving averages or by the sign and magnitude of recent returns. 

Liquidity regimes: High liquidity, low liquidity defined by bid-ask spreads or order book depth. 

Credit regimes: Risk-on, risk-off defined by credit spreads or cross-asset correlations.
For a concrete starting implementation, a three-state market regime model works well. Define:
State 0 as Bull: the 20-day return is above a positive threshold. 
State 1 as Bear: the 20-day return is below a negative threshold. 
State 2 as Sideways: everything in between.
The key requirement is that your states must be mutually exclusive and collectively exhaustive. Every observation must fall into exactly one state at any point in time. No gaps and no overlaps.
Once you have defined your states, you need to estimate the transition probabilities between them. This is the transition matrix. Each entry P(i,j) represents the probability of moving from state i to state j in one time step.
The transition matrix for a three-state model looks like this:
P = | P(0,0)  P(0,1)  P(0,2) | | P(1,0)  P(1,1)  P(1,2) | | P(2,0)  P(2,1)  P(2,2) |
Each row must sum to exactly 1.0 because from any given state, the system must transition to some state in the next step, including staying in the same state.
The maximum likelihood estimator for each transition probability is beautifully simple. Count how many times the system transitioned from state i to state j. Divide by the total number of transitions out of state i:
P̂(i,j) = Count(i → j) / Count(i → any state)
This is exactly how you would intuitively estimate a probability. It is also what the full derivation of the maximum likelihood estimator produces when you maximize the log-likelihood function for Markov Chain data. The intuitive answer and the mathematically rigorous answer are the same.

python
import numpy as np
import pandas as pd
import yfinance as yf

def define_market_states(returns, bull_threshold=0.02, bear_threshold=-0.02, window=20):
    rolling_return = returns.rolling(window).sum()
    states = np.where(rolling_return > bull_threshold, 0,
             np.where(rolling_return < bear_threshold, 1, 2))
    return pd.Series(states, index=returns.index)

def estimate_transition_matrix(states, n_states=3):
    transition_counts = np.zeros((n_states, n_states))
    state_values = states.dropna().values
    
    for i in range(len(state_values) - 1):
        current_state = int(state_values[i])
        next_state = int(state_values[i + 1])
        if not np.isnan(current_state) and not np.isnan(next_state):
            transition_counts[current_state][next_state] += 1
    
    transition_matrix = np.zeros((n_states, n_states))
    for i in range(n_states):
        row_sum = transition_counts[i].sum()
        if row_sum > 0:
            transition_matrix[i] = transition_counts[i] / row_sum
    
    return transition_matrix

ticker = yf.Ticker("SPY")
df = ticker.history(period="10y", interval="1d")
returns = df['Close'].pct_change().dropna()

states = define_market_states(returns)
P = estimate_transition_matrix(states)

state_names = ['Bull', 'Bear', 'Sideways']
print("Transition Matrix:")
print(pd.DataFrame(P, index=state_names, columns=state_names).round(3))
The transition matrix you produce from this process is the map of your market. Every entry tells you the probability of moving between two specific regimes. It is the foundation of everything that follows.
Part 3: Computing Multi-Step Transition Probabilities
This is where Markov Chains become genuinely powerful as a trading tool.
You now have the one-step transition matrix P. But what you really want to know as a trader is not just what happens next month. You want to know what the market is likely to look like in three months, six months, twelve months. You want to know the probability of starting in a bull regime today and ending up in a bear regime after twelve transitions.
This is where the Chapman-Kolmogorov equation comes in. It states that the n-step transition probability from state i to state j is the (i,j) entry of the matrix P raised to the nth power:
P^(n)(i,j) = [Pⁿ]ᵢⱼ
That is it. To compute the probability of transitioning from any state to any other state in n steps, you simply multiply the transition matrix by itself n times and read the corresponding entry.
This is an extraordinarily elegant result. All the mathematical complexity of computing paths through a state space over multiple steps reduces to a single matrix power.
python
def multi_step_transition(P, n_steps):
    return np.linalg.matrix_power(P, n_steps)

def regime_probability_forecast(P, current_state, n_steps, state_names):
    P_n = multi_step_transition(P, n_steps)
    probabilities = P_n[current_state]
    
    print(f"\nFrom {state_names[current_state]} regime:")
    print(f"After {n_steps} steps:")
    for i, (name, prob) in enumerate(zip(state_names, probabilities)):
        print(f"  Probability of {name}: {prob:.4f}")
    
    return probabilities

state_names = ['Bull', 'Bear', 'Sideways']

for steps in [1, 5, 12, 24]:

What this tells you as a trader is specific and actionable. If you are currently in a bull regime, you can now quantify the probability that you will still be in a bull regime in 12 months versus having transitioned to bear or sideways. That probability informs your position sizing, your hedging decisions, and your strategy allocation.
There is another critical output from this framework: the stationary distribution. As n becomes very large, the distribution across states converges to a fixed vector π regardless of the starting state:
π = π × P
The stationary distribution tells you the long-run proportion of time the market spends in each regime. In practice, you solve for π using:
python
def find_stationary_distribution(P):
    n = P.shape[0]
    A = (P.T - np.eye(n))
    A[-1] = 1
    b = np.zeros(n)
    b[-1] = 1
    
    pi = np.linalg.solve(A, b)
    return pi

pi = find_stationary_distribution(P)
state_names = ['Bull', 'Bear', 'Sideways']
print("\nStationary Distribution:")
for name, prob in zip(state_names, pi):
    print(f"  {name}: {prob:.4f}")
The stationary distribution is your long-run baseline. Any strategy that bets heavily on a regime that the stationary distribution tells you is rare is taking on significant tail risk. Knowing the long-run regime proportions is essential for strategy design.
Regardless of starting state, the probability distribution across regimes converges to a fixed stationary distribution as the number of steps increases.
Part 4: From Regime Model to Trading Signal
Having a regime model is not a trading strategy. Converting it into one requires connecting the regime probabilities to specific position decisions.
The core insight is this: the Markov Chain gives you a probability distribution over future states at each point in time. Your trading signal is a function of that distribution.
The simplest approach is direct regime-based allocation. When the model says you are in a bull regime, go long. When bear, go short or flat. When sideways, reduce position size.
A more sophisticated approach uses the full probability vector as input to position sizing. The current state distribution vector π_t represents your probability allocation across regimes at time t. You can construct a position that is proportional to your confidence in each regime:
python

def backtest_markov_strategy(returns, states, P, lookback=252):
    positions = []
    dates = []
    
    state_values = states.dropna()
    
    for i in range(lookback, len(state_values)):
        historical_states = state_values.iloc[i-lookback:i]
        P_estimated = estimate_transition_matrix(historical_states)
        
        current_state = int(state_values.iloc[i])
        position, probs = compute_trading_signal(current_state, P_estimated, None)
        
        positions.append(position)
        dates.append(state_values.index[i])
    
    positions_series = pd.Series(positions, index=dates)
    strategy_returns = positions_series.shift(1) * returns.loc[dates]
    
    sharpe = strategy_returns.mean() / strategy_returns.std() * np.sqrt(252)
    
    cumulative = (1 + strategy_returns).cumprod()
    rolling_max = cumulative.cummax()
    max_drawdown = ((cumulative - rolling_max) / rolling_max).min()
    
    print(f"Annualized Sharpe: {sharpe:.4f}")
    print(f"Maximum Drawdown: {max_drawdown:.4f}")
    print(f"Annual Return: {strategy_returns.mean() * 252:.4f}")
    
    return strategy_returns, positions_series
The walk-forward structure in the backtest above is critical. You re-estimate the transition matrix at each step using only the historical data available at that point. You never use future information to estimate past probabilities. This is the difference between a realistic backtest and one that is guaranteed to disappoint in live trading.
The walk-forward structure ensures the transition matrix is estimated only from data available at each point in time, preventing lookahead bias.

import numpy as np
import pandas as pd
import yfinance as yf
from sklearn.preprocessing import StandardScaler
import warnings
warnings.filterwarnings('ignore')

class MarkovChainTradingSystem:
    def __init__(self, n_states=3, lookback_window=252, regime_window=20):
        self.n_states = n_states
        self.lookback_window = lookback_window
        self.regime_window = regime_window
        self.transition_matrix = None
        self.state_names = ['Bull', 'Bear', 'Sideways']
    
    def fetch_data(self, ticker, period="10y"):
        data = yf.Ticker(ticker).history(period=period, interval="1d")
        self.returns = data['Close'].pct_change().dropna()
        self.prices = data['Close']
        return self.returns
    
    def label_states(self, returns, bull_thresh=0.02, bear_thresh=-0.02):
        rolling_ret = returns.rolling(self.regime_window).sum()
        states = np.where(rolling_ret > bull_thresh, 0,
                 np.where(rolling_ret < bear_thresh, 1, 2))
        return pd.Series(states, index=returns.index)
    
    def estimate_transition_matrix(self, states):
        counts = np.zeros((self.n_states, self.n_states))
        vals = states.dropna().values.astype(int)
        for i in range(len(vals) - 1):
            if 0 <= vals[i] < self.n_states and 0 <= vals[i+1] < self.n_states:
                counts[vals[i]][vals[i+1]] += 1
        P = np.zeros_like(counts)
        for i in range(self.n_states):
            if counts[i].sum() > 0:
                P[i] = counts[i] / counts[i].sum()
        return P
    
    def stationary_distribution(self, P):
        n = P.shape[0]
        A = (P.T - np.eye(n))
        A[-1] = 1
        b = np.zeros(n)
        b[-1] = 1
        try:
            return np.linalg.solve(A, b)
        except:
            return np.ones(n) / n
    
    def generate_signal(self, P, current_state):
        probs = P[current_state]
        return probs[0] - probs[1]
    
    def run_walkforward_backtest(self):
        states = self.label_states(self.returns)
        signals = []
        
        for i in range(self.lookback_window, len(states)):
            historical = states.iloc[i - self.lookback_window:i]
            P = self.estimate_transition_matrix(historical)
            current = int(states.iloc[i])
            signal = self.generate_signal(P, current)
            signals.append({'date': states.index[i], 'signal': signal,
                           'state': self.state_names[current]})
        
        signals_df = pd.DataFrame(signals).set_index('date')
        position = np.sign(signals_df['signal'])
        position[abs(signals_df['signal']) < 0.1] = 0
        
        strategy_returns = position.shift(1) * self.returns.loc[signals_df.index]
        
        sharpe = strategy_returns.mean() / strategy_returns.std() * np.sqrt(252)
        cumulative = (1 + strategy_returns).cumprod()
        max_dd = ((cumulative - cumulative.cummax()) / cumulative.cummax()).min()
        
        print(f"\nMarkov Chain Trading System Results:")
        print(f"Annualized Sharpe Ratio: {sharpe:.4f}")
        print(f"Maximum Drawdown: {max_dd:.4f}")
        print(f"Annualized Return: {strategy_returns.mean() * 252:.4f}")
        print(f"\nRegime Distribution:")
        print(signals_df['state'].value_counts(normalize=True).round(3))
        
        return strategy_returns, signals_df

The three assumptions you must understand before deploying:
The first is the Markov property itself. The model assumes that the next state depends only on the current state, not on longer history. In reality, markets sometimes exhibit longer-range dependencies. A trend that has been running for six months may have different transition probabilities than one that just started. You can partially address this by expanding your state space to include duration information, though this increases complexity significantly.
The second is time homogeneity. The model assumes transition probabilities are constant over time. They are not. The probability of a bull regime transitioning to bear was very different in 2008 than in 2021. The standard mitigation is rolling window re-estimation, which you have already seen in the walk-forward backtest above. Shorter windows adapt faster but produce noisier estimates. Longer windows produce more stable estimates but lag regime changes.
The third is sufficient data for reliable estimation. The maximum likelihood estimator converges to true transition probabilities as you observe more transitions. With few observations, particularly for rare transitions, your estimates will be noisy and unreliable. Always check that every cell in your transition matrix has been estimated from at least 20 to 30 observed transitions. If not, consider merging states or extending your data history.
Part 6: Hidden Markov Models - Taking the Framework Further
Every assumption in the model so far has one thing in common. You assumed you could see the regime.
You labeled each day as Bull, Bear, or Sideways using rolling returns. But the regime was never directly observable. You reverse engineered it from price. That is not the same thing. A bear regime that has not yet shown up in prices because institutional positioning is quietly shifting beneath the surface is completely invisible to your state labels.

This is the core limitation of observable Markov Chains. Hidden Markov Models fix it.
In a Hidden Markov Model, the true market regime is unobservable. Only noisy return observations are available, from which the hidden regime sequence must be inferred.
In an HMM, the true regime is a hidden state you cannot observe. What you can observe is the returns sequence. Each hidden state generates returns from its own distribution. The bull regime produces returns with a positive mean and low variance. The bear regime produces returns with a negative mean and high variance. The model learns both the regime transitions and the return distributions at the same time, from price data alone, without you ever labeling a single day by hand.
There are two algorithms that make this work.
The first is Baum-Welch. This is the algorithm that estimates all model parameters from the observable return sequence. You give it returns. It learns the transition matrix, the return distribution for each regime, and the starting probabilities, all without requiring labeled data. It runs forward through the sequence computing probabilities, then backward to refine them, repeating until convergence.
The second is Viterbi. Once you have a fitted model, Viterbi decodes the most likely sequence of hidden regimes that produced your observed returns. It does not give you soft probabilities. It gives you the single best path through the hidden state space. 
python
from hmmlearn import hmm
import numpy as np
import pandas as pd

def fit_market_hmm(returns, n_states=3, n_iter=1000):
    X = returns.values.reshape(-1, 1)
    
    model = hmm.GaussianHMM(
        n_components=n_states,
        covariance_type="full",
        n_iter=n_iter,
        random_state=42
    )
    model.fit(X)
    
    hidden_states = model.predict(X)
    
    state_means = {s: returns[hidden_states == s].mean() for s in range(n_states)}
    sorted_states = sorted(state_means, key=state_means.get, reverse=True)
    state_map = {old: new for new, old in enumerate(sorted_states)}
    labeled = np.array([state_map[s] for s in hidden_states])
    
    print("Learned Transition Matrix:")
    print(np.round(model.transmat_, 3))
    
    state_names = ['Bull', 'Bear', 'Sideways']
    for i in range(n_states):
        mask = labeled == i
        print(f"{state_names[i]}: mean={returns[mask].mean():.4f}, vol={returns[mask].std():.4f}")
    
    return model, labeled

model, labeled_states = fit_market_hmm(returns)
The signal generation is identical to Part 4. At each step you compute the probability of being in each regime right now. You multiply that vector by the transition matrix to get the one step ahead forecast. Bull probability minus bear probability gives your position.
python
def hmm_signal(model, returns, lookback=252):
    signals = []
    
    for i in range(lookback, len(returns)):
        window = returns.iloc[i - lookback:i].values.reshape(-1, 1)
        posteriors = model.predict_proba(window)
        current_probs = posteriors[-1]
        next_probs = current_probs @ model.transmat_
        signals.append(next_probs[0] - next_probs[1])
    
    return pd.Series(signals, index=returns.index[lookback:])
Two things to keep in mind before you deploy this.  

Baum-Welch finds a local maximum, not a global one. Always run it from multiple random starts and keep the model with the highest log likelihood. The default single initialization will frequently produce suboptimal regime assignments.

The emission variables you choose matter more than any other design decision. Returns alone carry limited information about the true economic regime. Returns combined with realized volatility, credit spreads, and VIX term structure give the model a much richer signal. The choice of what to feed as observations is where domain knowledge compounds the mathematics.
The observable Markov Chain gave you a regime map. The Hidden Markov Model builds that map in real time from noisy signals, without requiring a single manually labeled data point. The Summary
Markov Chains do not predict the future. What they do is something more useful. They quantify the probability of every possible future state given the current state. They give you a mathematical map of market regimes and the likelihood of transitioning between them.
The framework is fully implementable in a weekend. The transition matrix is estimated from historical data in a handful of lines of Python. The Chapman-Kolmogorov equation gives you n-step forecasts through a single matrix power. The stationary distribution tells you the long-run baseline. And the walk-forward backtest tells you whether your model has genuine predictive power or is fitting noise.
The assumptions are real and they matter. Time homogeneity is violated. The Markov property is an approximation. Estimation error is always present. def poisson_probability(lam, k):

def polymarket_poisson_edge(event_name, historical_rate, market_price_cents, 
                             time_periods=1):
    """
    Find edge on rare-event Polymarket contracts using Poisson distribution.
    
    event_name: description of the event
    historical_rate: average occurrences per time period (lambda)
    market_price_cents: current Polymarket price in cents
    time_periods: how many periods until expiry
    """
    # Scale lambda for the time period
    lam = historical_rate * time_periods
    
    # P(at least one event) = 1 - P(zero events)
    p_yes = 1 - poisson.pmf(0, lam)
    
    market_price = market_price_cents / 100
    edge_cents = (p_yes - market_price) * 100
    
    # Kelly fraction (quarter Kelly)
    if p_yes > market_price:
        b = (1 - market_price) / market_price  # net odds
        f_kelly = (b * p_yes - (1 - p_yes)) / b
        f_quarter = f_kelly * 0.25
        direction = "BUY YES"
    else:
        b = market_price / (1 - market_price)
        f_kelly = (b * (1 - p_yes) - p_yes) / b
        f_quarter = f_kelly * 0.25
        direction = "BUY NO"
    
    return {
        "event": event_name,
        "poisson_probability": p_yes,
        "market_price": market_price,
        "edge_cents": edge_cents,
        "quarter_kelly_fraction": max(0, f_quarter),
        "direction": direction
    }

import numpy as np

def shannon_entropy(p):
    """
    Calculate Shannon entropy in bits for a binary outcome.
    
    p: probability of YES (0 to 1)
    Returns: entropy in bits (0 to 1)
    """
    p = np.clip(p, 0.001, 0.999)
    q = 1 - p
    return -(p * np.log2(p) + q * np.log2(q))

def kl_divergence(p_mine, p_market):
    """
    Measure edge in bits. 
    
    p_mine: your probability estimate
    p_market: current market price
    Returns: KL divergence in bits
    
    Interpretation:
    < 0.05 bits: No edge. Skip.
    0.05-0.10 bits: Weak edge. Fees may eat it.
    0.10-0.20 bits: Real edge. Trade.
    > 0.30 bits: Check your math. Something is wrong.
    """
    p_mine = np.clip(p_mine, 0.01, 0.99)
    p_market = np.clip(p_market, 0.01, 0.99)
    
    q_mine = 1 - p_mine
    q_market = 1 - p_market
    
    return (p_mine * np.log2(p_mine / p_market) + 
            q_mine * np.log2(q_mine / q_market))

def scan_markets(my_estimates, market_prices, market_names):
    """
    Scan multiple markets, rank by KL-divergence edge.
    """
    results = []
    for name, p_mine, p_market in zip(market_names, my_estimates, market_prices):
        kl = kl_divergence(p_mine, p_market)
        edge_cents = (p_mine - p_market) * 100
        
        if kl < 0.05:
            signal = "SKIP — no edge"
        elif kl < 0.10:
            signal = "WEAK — fees may kill it"
        elif kl < 0.20:
            signal = "TRADE — real edge"
        else:
            signal = "RECHECK — suspiciously high"
            
        results.append({
            "market": name,
            "my_estimate": p_mine,
            "market_price": p_market,
            "edge_cents": edge_cents,
            "kl_bits": kl,
            "signal": signal
        })
    
    return sorted(results, key=lambda x: x["kl_bits"], reverse=True)

The Insider Detector
Shannon entropy does something else it detects insiders before news breaks.
A normal market drifts slowly. An insider moves differently: they know the outcome, open a large position, the price snaps in one direction, and entropy collapses sharply.
Watch entropy's rate of change. When it drops by 3 standard deviations without public news someone knows
python
import pandas as pd

def detect_insider_activity(price_history, window=12, threshold=3.0):
    """
    Detect abnormal entropy collapse that precedes news.
    
    price_history: list of (timestamp, price) tuples, 5-min intervals
    window: rolling window for baseline
    threshold: sigma threshold for alert (3 = 3-sigma rule)
    """
    df = pd.DataFrame(price_history, columns=['timestamp', 'price'])
    
    # Calculate entropy at each timestep
    df['entropy'] = df['price'].apply(shannon_entropy)
    
    # Rate of entropy change
    df['dH'] = df['entropy'].diff()
    
    # Rolling baseline
    df['dH_mean'] = df['dH'].rolling(window).mean()
    df['dH_std'] = df['dH'].rolling(window).std()
    
    # Alert when entropy drops abnormally fast
    df['z_score'] = (df['dH'] - df['dH_mean']) / df['dH_std']
    df['insider_alert'] = df['z_score'] < -threshold  # negative = entropy DROP
    
    alerts = df[df['insider_alert']].copy()
    
    return alerts[['timestamp', 'price', 'entropy', 'z_score']]

# Returns rows where entropy collapsed faster than 3-sigma baseline
# No public news at that moment = potential insider movement

How It Works on Polymarket
The Polymarket crowd prices events using intuitive Gaussian reasoning.
They see "Will Bitcoin drop 40% in a week?" and think: "That's a 4-sigma event. Almost impossible. 2¢."
A fat-tail trader thinks: "Bitcoin has experienced 40%+ drops in a week 3 times in 10 years. That's not a 2¢ probability. That's a 5-6¢ probability. The market is wrong by 3 cents on every contract."
3 cents doesn't sound like much. Across thousands of contracts, it's a systematic edge that compounds indefinitely.
python
import numpy as np
from scipy import stats
from scipy.stats import kurtosis, norm

def fat_tail_analysis(historical_returns, confidence=0.95):
    """
    Analyze whether a market's price history has fat tails
    and by how much Gaussian models underestimate extreme events.
    
    historical_returns: list of price changes per period
    confidence: VaR confidence level
    """
    returns = np.array(historical_returns)
    
    # Basic statistics
    mu = np.mean(returns)
    sigma = np.std(returns)
    
    # Kurtosis (normal distribution = 3, fat tails > 3)
    kurt = kurtosis(returns, fisher=False)  # Pearson's kurtosis
    excess_kurt = kurt - 3
    
    # Gaussian VaR vs actual VaR
    gaussian_var = norm.ppf(1 - confidence, mu, sigma)
    empirical_var = np.percentile(returns, (1 - confidence) * 100)
    
    # How much does Gaussian underestimate tail risk?
    tail_ratio = abs(empirical_var / gaussian_var) if gaussian_var != 0 else 1
    
    return {
        "kurtosis": kurt,
        "excess_kurtosis": excess_kurt,
        "fat_tails_present": kurt > 3,
        "gaussian_var_95": gaussian_var,
        "empirical_var_95": empirical_var,
        "gaussian_underestimates_by": f"{(tail_ratio - 1) * 100:.0f}%"
    }

def polymarket_fat_tail_edge(market_price_cents, gaussian_probability, 
                              fat_tail_multiplier):
    """
    Calculate edge when Polymarket prices assume Gaussian tails
    but reality has fat tails.
    
    market_price_cents: current Polymarket price
    gaussian_probability: what Gaussian model says
    fat_tail_multiplier: how much fatter the real tails are (from historical data)
    """
    true_probability = gaussian_probability * fat_tail_multiplier
    true_probability = min(true_probability, 0.99)
    
    market_price = market_price_cents / 100
    edge_cents = (true_probability - market_price) * 100
    
    return {
        "market_price": market_price,
        "gaussian_estimate": gaussian_probability,
        "fat_tail_estimate": true_probability,
        "edge_cents": edge_cents,
        "direction": "BUY YES" if edge_cents > 0 else "BUY NO"
    }

The Barbell Strategy on Polymarket
Taleb's practical framework isn't just about measuring fat tails. It's about positioning to survive and profit from events the market says can't happen.
The Barbell: put 85-90% of capital in maximum safety (high-probability, low-return markets near 90¢+), and 10-15% in extreme tail bets (markets at 2-5¢ that fat-tail math says should be 6-12¢).
python
def barbell_portfolio(bankroll, safe_markets, tail_markets):
    """
    Build a Taleb barbell on Polymarket.
    
    safe_markets: list of (price_cents, true_probability) for high-prob markets
    tail_markets: list of (price_cents, true_probability) for tail-risk markets
    """
    safe_allocation = bankroll * 0.85
    tail_allocation = bankroll * 0.15
    
    # Equal-weight within each bucket
    safe_per_market = safe_allocation / len(safe_markets)
    tail_per_market = tail_allocation / len(tail_markets)
    
    portfolio = {"safe": [], "tail": []}
    
    for price, true_prob in safe_markets:
        ev = (true_prob * (100 - price) - (1 - true_prob) * price) / price
        portfolio["safe"].append({
            "price": price,
            "true_prob": true_prob,
            "allocation": safe_per_market,
            "expected_ev": ev * safe_per_market
        })
    
    for price, true_prob in tail_markets:
        payout_multiple = (100 - price) / price  # e.g., 3¢ contract → 32x payout
        ev = true_prob * payout_multiple - (1 - true_prob)
        portfolio["tail"].append({
            "price": price,
            "true_prob": true_prob,
            "allocation": tail_per_market,
            "payout_multiple": payout_multiple,
            "expected_ev": ev * tail_per_market
        })
    
    return portfolio

The System: All Three Together
These three formulas attack the same problem from different angles.
Poisson tells you the true probability of rare events based on historical rates. Shannon tells you how much information edge you have and whether someone else has more. Taleb tells you where the crowd's probability model is structurally wrong.

def complete_rare_event_system(
    event_name,
    historical_rate,           # For Poisson
    market_price_cents,
    my_probability_estimate,   # For Shannon KL
    fat_tail_multiplier,       # For Taleb
    bankroll,
    days_to_expiry
):
    """
    Complete three-formula system for rare-event markets.
    """
    
    # STEP 1: POISSON — True base rate probability
    poisson_prob = 1 - np.exp(-historical_rate)
    
    # STEP 2: SHANNON — Measure information edge
    market_price = market_price_cents / 100
    kl = kl_divergence(my_probability_estimate, market_price)
    
    # STEP 3: TALEB — Fat tail adjustment
    gaussian_prob = norm.sf(2.0)  # Baseline 2-sigma assumption
    fat_tail_prob = min(gaussian_prob * fat_tail_multiplier, 0.99)
    
    # Synthesize: weighted average of three estimates
    final_probability = (
        0.40 * poisson_prob +
        0.35 * my_probability_estimate +
        0.25 * fat_tail_prob
    )
    
    edge_cents = (final_probability - market_price) * 100
    
    # Kelly sizing
    if final_probability > market_price:
        b = (1 - market_price) / market_price
        f_kelly = max(0, (b * final_probability - (1 - final_probability)) / b)
        f_quarter = f_kelly * 0.25
        direction = "BUY YES"
    else:
        direction = "PASS"
        f_quarter = 0
    
    return {
        "event": event_name,
        "poisson_estimate": poisson_prob,
        "my_estimate": my_probability_estimate,
        "fat_tail_estimate": fat_tail_prob,
        "final_probability": final_probability,
        "market_price": market_price,
        "edge_cents": edge_cents,
        "kl_edge_bits": kl,
        "direction": direction,
        "quarter_kelly_pct": f_quarter,
        "position_size_usd": bankroll * f_quarter
    }

    while True:

            "active":   "true",
            "closed":   "false",
            "limit":    GAMMA_PAGE_LIMIT,

// Rust cursor-based scanner — CLOB V2
async fn fetch_all_markets(client: &Client, args: &Args) -> Result<Vec<GammaMarket>> {
    let mut all_markets: Vec<GammaMarket> = Vec::new();
    let mut next_cursor: Option<String> = None;

    loop {
        let mut params = vec![
            ("active",  "true"),
            ("closed",  "false"),
            ("limit",   "100"),
        ];
        let cursor_val;
        if let Some(ref c) = next_cursor {
            cursor_val = c.clone();
            params.push(("next_cursor", &cursor_val));
        }

        let resp = client
            .get(&format!("{}/markets/keyset", GAMMA_BASE_URL))
            .query(¶ms).send().await?
            .json::<KeysetResponse>().await?;

        all_markets.extend(resp.markets);
        next_cursor = resp.next_cursor;
        if next_cursor.is_none() { break; }

        tokio::time::sleep(Duration::from_millis(200)).await;
    }
    Ok(all_markets)
}

Reflexivity (The Edge of the Map)
Most traders think price reflects probability.
Actually, not always.
At the deepest level of the market, something stranger happens - price itself starts to influence probability. The act of trading changes the thing being traded. The map redraws the territory.
This is reflexivity - a concept George Soros built a $40 billion fortune on, and one almost no Polymarket trader has internalised.
In a perfectly efficient market, price is a passive observer. It reflects underlying reality without distorting it. In a reflexive market, price is an active participant. It feeds back into the very probabilities it's supposed to be measuring.
On Polymarket, reflexivity is everywhere. And it's where the deepest edges live.
The mechanism:
Reflexivity emerges when three conditions overlap on a market:
1. Visible price movement (the chart is being watched)
2. Thin liquidity (small volume moves price meaningfully)
3. Narrative attachment (price is interpreted as evidence)
When all three converge, a feedback loop forms. A price move generates attention. Attention generates trades. Trades move the price further. The original move - which may have been pure noise - gets retroactively justified by the very market response it triggered.

Three structural features of Polymarket make reflexivity unusually strong:

The execution framework:
Quants approach reflexivity in two opposite ways depending on the phase of the loop:
Phase 1 - Riding the loop (early)
When a reflexive cascade begins, the trade is to enter early and exit before the loop saturates. The signal is:

Coincident X amplification (post crosses 5K impressions)
No meaningful underlying news to anchor the move
In this phase, you're not trading the event. You're trading the loop itself.
Phase 2 - Fading the break (late)
Every reflexive loop terminates. When the loop breaks, price reverts toward the pre-loop equilibrium with surprising speed. The signals that the loop is exhausting:
New entrants slowing (tx velocity peaking)
Spread starting to widen (LPs sensing risk)
Narrative becoming explicit ("everyone agrees this is going to 90¢")
When the consensus becomes the trade, the trade is over.

Why Machine Learning in Trading
Classical strategies have one great strength: they are understandable. You know exactly why the algorithm entered a position. But they have a fundamental limitation: the rules are invented by a human. And humans can miss patterns that are not obvious to the eye.
The ML approach inverts the logic: we do not tell the model what to look for. We hand it data and say: "Find whatever predicts price movement."
This opens up new possibilities, but also new risks. The main one is overfitting: the model memorizes noise instead of signal. We will cover how to fight that.
Three Tasks, Three Approaches
In trading, ML is used for three types of problems:
- **Regression**: predict a numerical value - tomorrow's price or return
- **Classification**: predict direction - will the price go up or down (0 or 1)
- **Anomaly detection and clustering**: find unusual market regimes
In this article we cover the first two. The third will come later.

Feature Engineering
Feature engineering is a critical step in ML for trading. The model does not work with the raw price - it works with features we extract from it. The quality of the features often matters more than the choice of model.

python
def make_features(df, lags=5):
    """
    Build features from lagged log returns.

    Parameters:
        df   -- DataFrame with a 'log_return' column
        lags -- number of lags

    Returns:
        DataFrame with features and target variable
    """
    data = df[["log_return"]].copy()

    # Lags: returns from past days
    for lag in range(1, lags + 1):
        data[f"lag_{lag}"] = data["log_return"].shift(lag)

    # Regression target: tomorrow's return
    data["target_return"] = data["log_return"].shift(-1)

    # Classification target: direction (1 = up, 0 = down)
    data["target_direction"] = (data["target_return"] > 0).astype(int)

    # Drop rows with NaN
    data.dropna(inplace=True)

    return data

data = make_features(df, lags=5)

print(data.head())
print(f"\nDataset shape: {data.shape}")
print(f"Fraction of up days: {data['target_direction'].mean():.1%}")

The idea is simple: if BTC has been rising for the last 5 days, it might keep rising tomorrow (momentum) - or pull back (mean reversion). The model will figure out which effect is actually at work.

Backtesting the ML Strategy
Now let us turn the model's predictions into trading signals and run a backtest.

def ml_backtest(data, y_pred_cls, test_index):
    """
    Simple backtest of an ML strategy.

    Parameters:
        data       -- original DataFrame with log_return
        y_pred_cls -- predicted classes (0 or 1) on the test period
        test_index -- index of the test set

    Returns:
        result -- DataFrame with results
    """
    result = data.loc[test_index].copy()

    # Position: 1 if the model predicts up, 0 if down
    result["position"] = y_pred_cls
    # Shift by 1 day (we trade the next day)
    result["position"] = result["position"].shift(1)

    result.dropna(subset=["position"], inplace=True)

    # Strategy return
    result["strategy_return"]   = result["log_return"] * result["position"]

    # Cumulative returns
    result["cumret_strategy"]   = result["strategy_return"].cumsum().apply(np.exp)
    result["cumret_buyandhold"] = result["log_return"].cumsum().apply(np.exp)

    return result

result = ml_backtest(data, y_pred_cls, X_test.index)

# Metrics

bh_return     = result["cumret_buyandhold"].iloc[-1] - 1
n             = len(result)
annual_return = (1 + total_return) ** (252 / n) - 1
annual_vol    = result["strategy_return"].std() * np.sqrt(252)
sharpe        = annual_return / annual_vol if annual_vol > 0 else 0

rolling_max   = result["cumret_strategy"].cummax()
max_drawdown  = ((result["cumret_strategy"] - rolling_max) / rolling_max).min()

print(f"Strategy return:   {total_return:+.1%}")
print(f"Buy-and-hold:      {bh_return:+.1%}")

def make_rich_features(df, lags=5):
    """
    Extended feature engineering with technical indicators.
    """
    data = df[["close", "volume", "log_return"]].copy()

    # Return lags
    for lag in range(1, lags + 1):
        data[f"lag_{lag}"] = data["log_return"].shift(lag)

    # Moving average ratio
    data["sma_20"] = data["close"].rolling(20).mean()
    data["sma_50"] = data["close"].rolling(50).mean()
    data["sma_ratio"] = data["sma_20"] / data["sma_50"] - 1  # >0: bullish signal

    # Momentum
    data["momentum_10"] = np.log(data["close"] / data["close"].shift(10))
    data["momentum_20"] = np.log(data["close"] / data["close"].shift(20))

    # Realized volatility over the last 10 days
    data["volatility_10"] = data["log_return"].rolling(10).std()

    # Z-score of price relative to 20-day mean
    rolling_mean = data["close"].rolling(20).mean()
    rolling_std  = data["close"].rolling(20).std()
    data["zscore_20"] = (data["close"] - rolling_mean) / rolling_std

    # Relative volume
    data["volume_ratio"] = data["volume"] / data["volume"].rolling(20).mean()

    # Target
    data["target_direction"] = (data["log_return"].shift(-1) > 0).astype(int)

    data.dropna(inplace=True)

    return data

A Word on Overfitting
The more expressive the model, the higher the risk of memorizing the training data. A random forest with 500 trees will show 99% accuracy on train and 51% on test. That is a textbook example of overfitting.
Signs of overfitting:
- A large gap between train and test accuracy
- The model performs below baseline on test
- Adding more features improves train but not test
How to fight it: fewer parameters, regularization, cross-validation, walk-forward testing.

 Walk-Forward Validation for ML
In Article 6 we covered walk-forward analysis for classical strategies. For ML it is even more important. A simple two-chunk train/test split is an overly optimistic estimate of real performance.
Walk-forward for ML: a rolling training window, with predictions made one step at a time.
python
def walk_forward_ml(data, feature_cols, target_col, train_size=252, step=21):
    """
    Walk-forward validation of an ML classifier.

    Parameters:
        data         -- DataFrame with features and target
        feature_cols -- list of feature column names
        target_col   -- name of the target column
        train_size   -- training window size (days)
        step         -- rolling step (days)

    Returns:
        DataFrame with predictions and actual values
    """
    X    = data[feature_cols].values
    y    = data[target_col].values
    idx  = data.index

    all_preds  = []
    all_actual = []
    all_dates  = []

    for start in range(0, len(data) - train_size - step, step):
        end_train = start + train_size
        end_test  = end_train + step

        X_tr, y_tr = X[start:end_train], y[start:end_train]
        X_te, y_te = X[end_train:end_test], y[end_train:end_test]

        sc   = StandardScaler()
        X_tr = sc.fit_transform(X_tr)
        X_te = sc.transform(X_te)

        clf  = LogisticRegression(random_state=42, max_iter=500)
        clf.fit(X_tr, y_tr)
        preds = clf.predict(X_te)

        all_preds.extend(preds)
        all_actual.extend(y_te)
        all_dates.extend(idx[end_train:end_test])

    wf_result = pd.DataFrame({
        "date":      all_dates,
        "actual":    all_actual,
        "predicted": all_preds,
    }).set_index("date")

    wf_acc = accuracy_score(wf_result["actual"], wf_result["predicted"])
    print(f"Walk-forward accuracy: {wf_acc:.1%}")

    return wf_result

wf = walk_forward_ml(data_rich, feature_cols_rich, "target_direction",
                     train_size=252, step=21)

Walk-forward accuracy is typically lower than accuracy on a fixed test set. That is the honest estimate of what to expect in live trading.

from sklearn.metrics import accuracy_score

def make_features(df, lags=5):
    data = df[["close", "volume", "log_return"]].copy()
    for lag in range(1, lags + 1):
        data[f"lag_{lag}"] = data["log_return"].shift(lag)
    data["sma_ratio"]     = df["close"].rolling(20).mean() / df["close"].rolling(50).mean() - 1
    data["momentum_10"]   = np.log(df["close"] / df["close"].shift(10))
    data["momentum_20"]   = np.log(df["close"] / df["close"].shift(20))
    data["volatility_10"] = data["log_return"].rolling(10).std()
    rm = df["close"].rolling(20).mean()
    rs = df["close"].rolling(20).std()
    data["zscore_20"]     = (df["close"] - rm) / rs
    data["volume_ratio"]  = df["volume"] / df["volume"].rolling(20).mean()
    data["target"]        = (data["log_return"].shift(-1) > 0).astype(int)
    data.dropna(inplace=True)
    return data


def run_ml_backtest(data, feature_cols, train_ratio=0.7):
    split_idx = int(len(data) * train_ratio)
    X_train = data[feature_cols].iloc[:split_idx]
    X_test  = data[feature_cols].iloc[split_idx:]
    y_train = data["target"].iloc[:split_idx]
    y_test  = data["target"].iloc[split_idx:]

    sc     = StandardScaler()
    Xtr_sc = sc.fit_transform(X_train)
    Xte_sc = sc.transform(X_test)

    clf    = LogisticRegression(random_state=42, max_iter=1000)
    clf.fit(Xtr_sc, y_train)
    preds  = clf.predict(Xte_sc)

    acc    = accuracy_score(y_test, preds)

    result = data.iloc[split_idx:].copy()
    result["position"]        = pd.Series(preds, index=X_test.index).shift(1)
    result.dropna(subset=["position"], inplace=True)
    result["strategy_return"] = result["log_return"] * result["position"]
    result["cumret_strategy"] = result["strategy_return"].cumsum().apply(np.exp)
    result["cumret_bh"]       = result["log_return"].cumsum().apply(np.exp)

    total_ret  = result["cumret_strategy"].iloc[-1] - 1
    annual_vol = result["strategy_return"].std() * np.sqrt(252)
    annual_ret = (1 + total_ret) ** (252 / len(result)) - 1
    sharpe     = annual_ret / annual_vol if annual_vol > 0 else 0
    rm         = result["cumret_strategy"].cummax()
    max_dd     = ((result["cumret_strategy"] - rm) / rm).min()

    print(f"Accuracy:        {acc:.1%}")
    print(f"Baseline:        {y_test.mean():.1%}")
    print(f"Strategy return: {total_ret:+.1%}")
    print(f"Buy & Hold:      {result['cumret_bh'].iloc[-1] - 1:+.1%}")
    print(f"Sharpe ratio:    {sharpe:.2f}")
    print(f"Max drawdown:    {max_dd:.1%}")

    return result, acc

Summary
In this article, we made a meaningful shift from hand-crafted rules to trained models:
- Built a feature engineering pipeline: lags, momentum, volatility, z-score, volume
- Applied linear regression to return prediction and interpreted the result honestly (R² near zero is normal)
- Moved to classification with logistic regression and achieved accuracy above baseline
- Compared several classifiers on the same dataset
- Built an equity curve for the ML strategy and computed the Sharpe ratio
- Covered feature importance and walk-forward validation as defenses against overfitting
The main takeaway: ML in trading works, but modestly. Accuracy of 54–57% is a realistic, working result - not the 99% you see in textbook examples. A signal that small, combined with proper position sizing and controlled transaction costs, can still form the basis of a profitable strategy.

Content : Different markets. Same edge. Why weather and 5-minute markets are the same game
At the core of most profitable strategies on Polymarket lies the same mathematical logic: the bot checks whether there is an edge, and only then decides whether to enter a trade. 
Two market types that look completely different on the surface — weather outcomes and 5-minute BTC windows — are actually the same game under the hood.
Foundation: Expected Value (EV)
The simplest form of EV answers one question: if I make this trade thousands of times, do I come out ahead?
EV = q − p
q — your estimate of the true probability (what you believe)
p — the current market price (what the crowd believes)
If q = 0.62 and p = 0.55, then EV = +0.07. That's a 7-cent edge per dollar risked — trade it.
The full payout formula
The simplified form hides the mechanics. The complete version prices both outcomes explicitly:
EV = q(1 − p) − (1 − q)p
Example. Market price p = 0.40, your model says q = 0.60:
EV = 0.60 × (1 − 0.40) − (1 − 0.60) × 0.40
= 0.60 × 0.60 − 0.40 × 0.40
= 0.36 − 0.16
= +0.20
That's 20 cents of edge per dollar. Strong signal to trade. Flip it — if q = 0.35 and p = 0.40:
EV = 0.35 × 0.60 − 0.65 × 0.40
= 0.21 − 0.26
= −0.05
Negative EV. Skip the trade.
EV > 0 doesn't guarantee profit on a single trade — only over many repetitions. A +0.20 EV trade still loses 40% of the time.
Position sizing: Kelly criterion
Knowing you have edge isn't enough. How much of your bankroll do you bet?
f* = (q − p) / (1 − p)
f* — fraction of bankroll to deploy
q — your probability
p — market price
Example. q = 0.60, p = 0.40:
f* = (0.60 − 0.40) / (1 − 0.40)
= 0.20 / 0.60
= 0.333  →  33.3% of bankroll
Now try a weaker edge: q = 0.45, p = 0.40:
f* = (0.45 − 0.40) / (1 − 0.40)
= 0.05 / 0.60
= 0.083  →  8.3% of bankroll
Same market price, but less confidence — Kelly automatically scales position size down to 8%. The formula is self-regulating.
Why bots use fractional Kelly
Full Kelly maximises long-run growth in theory, but in practice:
Your q estimate is never perfectly accurate
Model error compounds aggressively at full Kelly
Drawdowns at full Kelly are psychologically brutal even when EV is positive
Most systematic traders use ¼ to ½ Kelly. At the 0.60/0.40 example above, ½ Kelly gives f* ≈ 16.7% — still meaningful exposure, but far more robust to estimation error.
Where the edge comes from
Edge = q_real − p_market
The market price p_market reflects the crowd's current belief. Your edge exists when real-world data has already shifted the true probability, but the market hasn't caught up yet.
Concrete example (weather market). A weather station in Phoenix shows the temperature crossing 104°F at 11:47 AM. The Wunderground report that resolves the market won't update until 12:05 PM. From 11:47 to 12:05, the market is still pricing the "over 104°F" outcome at p = 0.55 — but a bot reading the raw sensor knows q_real ≈ 0.95. That's a 40-point edge that exists for 18 minutes.
Concrete example (5-min BTC market). The Chainlink oracle that resolves the market aggregates price data with a lag behind spot exchanges. A bot watching raw exchange feeds knows BTC just printed $76,420 — but the oracle is still reflecting $76,380. The resolution price is nearly certain, and the market hasn't adjusted yet.
The edge exists because most market participants are humans who react late to new data, sleep during low-liquidity windows, and don't monitor micro-updates in niche markets.
Intra-market arbitrage
Occasionally, the market misprices both sides simultaneously:
P(Yes) + P(No) < 1
In any binary market, the two sides should sum to approximately 1. When the sum falls below 1, you can buy both sides and lock in a guaranteed profit regardless of outcome.
Example. You observe P(Yes) = 0.46, P(No) = 0.50:
Total cost = 0.46 + 0.50 = 0.96
Guaranteed payout = 1.00
Guaranteed profit = 0.04 per dollar pair
Small, but risk-free and scalable across hundreds of positions. These windows are narrow and close quickly — which is why automated bots are the primary extractors of this alpha.
Near-expiry scalping
A separate strategy from arbitrage — but equally systematic.
The idea is simple: buy a YES or NO outcome at 95–99¢ when the result is already virtually certain, and collect the remaining 1–5¢ as the market resolves to $1.00.
Example 1 — weather market. With 8 minutes left, the ASOS station feed shows 109.4°F. The "above 108°F" outcome is priced at p = 0.96. Resolution is near-certain. The bot buys at 0.96 and collects 0.04 on resolution.
EV per dollar = 1.00 − 0.96 = +0.04
Example 2 — 5-minute BTC market. With 30 seconds left in the window, BTC spot is at $76,850. The "above $76,400 at close" outcome is priced at p = 0.97. The bot buys at 0.97 and collects 0.03 on resolution.
EV per dollar = 1.00 − 0.97 = +0.03
Small per trade. But at high volume — dozens of positions simultaneously across multiple markets — it compounds fast.
Why it works:
Most participants won't touch a 96¢ outcome — the upside looks too thin
The bot doesn't care about thin margins, it cares about certainty-adjusted volume
Liquidity is often available because human traders exit early, leaving mispriced near-certain outcomes on the table
The risk: a 99¢ outcome flipping to 0 is rare when entering close to expiration — but it happens. One failed trade can wipe the gains from the previous 99. This is why bots don't enter blindly: if volatility is high, the trade is skipped entirely. The edge only exists when the outcome is genuinely near-certain, not just priced that way.
This is not the same as oracle lag arbitrage. It's a separate edge: pricing inefficiency near expiration, not data delay.
Bayesian updating: probability in motion
A static probability estimate is fine for a single moment. But data keeps arriving. Bayes' theorem describes how to update correctly:
P(A|B) = [P(B|A) × P(A)] / P(B)
Example. Market: "Chicago max temperature above 90°F today" — p = 0.40. Prior based on July climate statistics: P(>90°F) = 0.35. Then at 11:23 AM the O'Hare station records 88.4°F, and you see in the raw ASOS feed that temperature is rising at ~1.2°F per hour. Five hours remain until resolution (final daily high per Wunderground). At this rate, the probability of breaking 90°F is around 78%.
Bayesian update: P(this growth rate | outcome >90°F) = 0.85, base rate of this growth pattern on any day: P(rate) = 0.39.
P(>90°F | growth rate) = (0.85 × 0.35) / 0.39
= 0.2975 / 0.39
= 0.763
Probability updated from 35% → 76%. Market is still at p = 0.40 — edge is +36¢. This is exactly what sophisticated weather bots do in real time: not weather forecasting, but arbitrage between a raw sensor reading and the market price.
Markov Chains: modeling state transitions
For markets with sequential structure — like rolling 5-minute windows — Markov chains model how the current state predicts the next one:
P(Xₜ₊₁ = j | Xₜ = i) = Pᵢⱼ
Xₜ — current state (e.g. BTC price above/below threshold right now)
Xₜ₊₁ — next state at resolution
Pᵢⱼ — probability of transitioning from state i to state j
Example. A simple 2-state model for 5-min BTC windows:
If BTC is currently above the threshold with 4 minutes left in the window, the model gives 72% probability it stays above at resolution. If the market prices this at p = 0.60, your edge is +12¢.
Markov models are not central to most Polymarket bots today, but become useful once you accumulate enough historical data on a specific market's transition behavior.
How this works on 5-minute BTC markets
Resolution is based on Chainlink data. The bot is not predicting direction — it's trading the value at a specific moment in time (start vs. end of the 5-minute window).
Key exploitable factors:
Oracle delay: Chainlink aggregates from multiple sources with a 3–7 second lag behind spot exchanges
Desynchronization: the market price reflects trader expectations, not the oracle's current reading
Low-liquidity windows: 2–5 AM UTC when fewer humans are watching, spreads are wider, and mispricing persists longer
Numeric example. At 03:14 UTC, BTC spot is $69,450. The Chainlink oracle feed shows $69,388 (62 seconds old). The "BTC above $69,400 at close" market is priced at p = 0.48. A bot with a direct exchange feed computes q ≈ 0.87 based on current spot.
EV = 0.87 × (1 − 0.48) − (1 − 0.87) × 0.48
= 0.87 × 0.52 − 0.13 × 0.48
= 0.452 − 0.062
= +0.39
39 cents of edge per dollar. Kelly formula gives f* = (0.87 − 0.48) / (1 − 0.48) = 0.75 — but full Kelly is not applicable in practice.
Real bots with PnL +$100,000 size positions very differently:
One-sided position: $10–$1,000 per trade
Arbitrage pair (both sides): up to $3,000
The reasons: q is never exactly 0.87 — it's an estimate with embedded model error. High-frequency strategy runs on volume: hundreds of trades per day at small size produce more stable results than rare large bets. Liquidity in niche markets is limited — a large order moves the price and destroys the edge itself. A single bad estimate at large size can erase the profit from dozens of correct trades.
The real position sizing logic in these bots is closer to a fixed size scaled by edge width, not a percentage of bankroll per Kelly.

How this works in weather markets
Resolution is based on Wunderground data. The bot is not forecasting weather — it's trading what value will appear in the final report.
Key exploitable factors:
Update cadence: stations update at irregular intervals, not every minute
Finalization timing: the "official" value often lags real conditions by 10–20 minutes
Revision windows: some stations allow late corrections — knowing when data becomes final is alpha

EV = 0.92 × 0.65 − 0.08 × 0.35

57 cents of edge. This is why a single correct trade in a weather market can be worth more than 20 directional BTC trades.

Why a low win rate can still be profitable
This confuses most observers who look at bot trade history. The math:
Total EV across N trades:
Total EV = Σ [qᵢ(1 − pᵢ) − (1 − qᵢ)pᵢ]
What matters is the sum, not the count of wins.
Example. A bot makes 100 trades:
70 trades: small losses averaging −$2 each → −$140
30 trades: large wins averaging +$12 each → +$360
Net: +$220 on 100 trades, despite a 30% win rate
This is the asymmetry model: deliberately accept many small losses to harvest occasional large payouts when the oracle lag creates a wide edge. Not guessing — manufacturing asymmetric payoffs through data advantages.
The core insight
You are not trading the event itself. You are trading the gap between real-world data and how it is reflected in the market price — and specifically, the timing of that gap's closure.
The winners understand three things:
Where the data comes from — which oracle, which station, which API endpoint
How it updates — cadence, aggregation method, source weighting
When it becomes final — the exact moment q converges to 0 or 1
On the surface: a stream of profitable trades and growing PnL. In reality: bots systematically harvesting the premium that exists between the physical world and its digital reflection in a prediction market.

import numpy as np

def shannon_entropy(p):
    """Returns entropy in bits for a binary outcome."""
    if p <= 0 or p >= 1:
        return 0.0
    q = 1 - p
    return -(p * np.log2(p) + q * np.log2(q))

import numpy as np

def kl_divergence_binary(p_mine, p_market):
    """Returns KL divergence in bits for a binary market.
       p_mine   — your estimate (0..1)
       p_market — market price (0..1)
    """
    # Clip to avoid log(0)
    p_mine   = np.clip(p_mine,   0.01, 0.99)
    p_market = np.clip(p_market, 0.01, 0.99)

    q, qm = 1 - p_mine, 1 - p_market
    return (p_mine * np.log2(p_mine / p_market)
            + q * np.log2(q / qm))

# Your estimate 0.55, market price 0.35
edge = kl_divergence_binary(0.55, 0.35)
print(f"Edge: {edge:.3f} bits")  # 0.120 bits

Averaging - throws away information, weights all signals equally even if one dominates.

import numpy as np
from scipy.optimize import minimize

def max_entropy_fusion(signals, confidences):
    """
    signals     — list of probability estimates [p1, p2, ..., pN]
    confidences — list of weights (higher = more trust)
    Returns a single fused probability via max-entropy weighting.
    """
    signals     = np.array(signals)
    confidences = np.array(confidences)

    # Constraint: weighted mean matches signal consensus
    target = np.sum(signals * confidences) / np.sum(confidences)

    # Max-entropy under mean constraint on {0,1} = Bernoulli(target)
    return np.clip(target, 0.01, 0.99)

signals     = [0.40, 0.60, 0.50]
confidences = [1.0,  2.0,  1.5]  # news most trusted

fused = max_entropy_fusion(signals, confidences)
print(f"Fused probability: {fused:.3f}")  # ~0.522

PART 3 - THE LSTM ARCHITECTURE
Market data is sequential. What happened in the last five minutes is connected to what happens next through autocorrelation, momentum, mean reversion, and microstructure dynamics.
A standard feedforward network treats every input independently. When you present it with today's feature vector, it has no knowledge of what the last 50 days looked like. For time series data, this throws away the most important information in your datase

import torch
import torch.nn as nn

class TradingLSTM(nn.Module):
def __init__(self, input_size, hidden_size=64,
num_layers=2, dropout=0.2):
super().__init__()
self.hidden_size = hidden_size
self.num_layers = num_layers
self.lstm = nn.LSTM(
input_size=input_size,
hidden_size=hidden_size,
num_layers=num_layers,
dropout=dropout,
batch_first=True
)
self.dropout = nn.Dropout(dropout)
self.fc = nn.Linear(hidden_size, 1)
self.sigmoid = nn.Sigmoid()

def forward(self, x):
h0 = torch.zeros(self.num_layers, x.size(0), self.hidden_size)
c0 = torch.zeros(self.num_layers, x.size(0), self.hidden_size)
out, _ = self.lstm(x, (h0, c0))
return self.sigmoid(self.fc(self.dropout(out[:, -1, :])))
The lookback window - how many time steps you feed into the model at once - is a critical hyperparameter. For daily strategies, start with 10 to 20 trading days. For intraday strategies on 5-minute bars, 24 periods covers two hours of context. Determine the optimal window empirically through validation performance. Not by intuition.
PART 4 - TRAINING WITHOUT DECEIVING YOURSELF
Building the model is easy. Evaluating honestly whether it learned something real is where almost everyone fails.
Overfitting is the central failure mode. An overfit model found patterns in its training data that don't generalize to new data. On training data it performs perfectly. On new data it performs at chance or worse. The problem: overfit in-sample performance looks identical to genuine edge. You cannot distinguish them by looking at training metrics alone.
The solution is a strict three-way sequential data split.
Training set is where gradient descent runs. The model sees this data repeatedly and adjusts parameters based on it. Never evaluate generalization on this set.
Validation set is data the model never trains on. Monitor validation loss after every epoch. The moment validation loss stops improving and begins increasing while training loss continues falling - you've found the overfitting threshold. Stop training immediately. Save the weights from the epoch with the lowest validation loss. This is early stopping.
Test set is data the model has never influenced in any form. Use it exactly once - after all architecture decisions, feature engineering decisions, and hyperparameter choices are finalized using the training and validation sets. The test set result is your honest estimate of live performance. Using it to make additional design decisions invalidates it permanently.
These splits must be sequential in time. The training period comes first. The validation period follows. The test period is most recent. Randomizing the split would allow future information to contaminate training - a form of lookahead bias that guarantees your backtest looks better than live performance.
def train_model(model, train_loader, val_loader, patience=10):
optimizer = torch.optim.Adam(model.parameters(), lr=0.001)
criterion = nn.BCELoss()
best_val_loss = float('inf')
best_weights = None
patience_counter = 0

for epoch in range(100):
model.train()
for X_batch, y_batch in train_loader:
optimizer.zero_grad()
loss = criterion(model(X_batch).squeeze(), y_batch)
loss.backward()
torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
optimizer.step()

model.eval()
val_loss = sum(
criterion(model(X).squeeze(), y).item()
for X, y in val_loader
) / len(val_loader)

if val_loss < best_val_loss:
best_val_loss = val_loss
best_weights = model.state_dict().copy()
patience_counter = 0
else:
patience_counter += 1
if patience_counter >= patience:
print(f"Early stopping at epoch {epoch}")
break

model.load_state_dict(best_weights)
return model
Gradient clipping (clip_grad_norm_ with max_norm=1.0) is included for a specific reason. LSTM gradients can explode during training when learned sequences are long. Clipping the gradient norm prevents this without significantly affecting convergence.
